# Release 1.1.1

## Packages

| Package      | Version |

## Highlights

This is a bug-fix release for `rxfy-react`. It fixes `useStateData` latching a spurious `REJECTED` state (surfacing an `AbortError`) when a component unmounts before its initial fetch settles — most visibly under React StrictMode, where the synchronous mount → unmount → mount cycle aborted the in-flight fetch and the remount never refetched. `rxfy` is republished at the same version with no source changes.

## Changes

### Bug Fixes

- **`useStateData` no longer drops a fetch on a transient unmount** — the client fetch is now started as a subscribe-time side effect and multicast via RxJS `share()` with a deferred `resetOnRefCountZero` (`timer(0)`). A synchronous unsubscribe → resubscribe (a StrictMode remount, or one of several subscribers leaving) cancels the pending ref-count reset, so the in-flight request survives instead of being torn down. Abort-driven rejections are also guarded by the request's `AbortSignal`, so they no longer write a spurious result or `REJECTED` into the shared query atom; a genuine teardown rolls an unsettled query back to `IDLE` so the next subscriber refetches. SSR fetches are unaffected — they omit the signal, preserving the original behavior. ([`packages/rxfy-react/src/useStateData.ts`](../../packages/rxfy-react/src/useStateData.ts))
