# Release 2.0.0

## Packages

| Package         | Version |
| --------------- | ------- |
| `rxfy`          | `2.0.0` |
| `rxfy-react`    | `2.0.0` |
| `rxfy-protocol` | `2.0.0` |
| `rxfy-server`   | `2.0.0` |
| `rxfy-ws`       | `2.0.0` |

## Highlights

rxfy 2.0 turns the reactive store into a full live-app stack. Three new packages — `rxfy-protocol` (wire contract), `rxfy-server` (Drizzle-bound resources, writes, and capability grants), and `rxfy-ws` (WebSocket transport) — deliver real-time entity updates from the database to the client, and `rxfy-react` gains a live client that surfaces per-state "updates available" signals through `useStateData`. All five packages now version in lockstep.

## Changes

### New Features

- **`rxfy-protocol` (new package)** — the standalone wire contract for rxfy live updates: `ServerMessage`/`ClientMessage` types, `PROTOCOL_VERSION`, message constructors, and a `serialize`/`parseServerMessage`/`parseClientMessage` codec (superjson-based, so `Date` and other rich types survive the wire). See `packages/rxfy-protocol/src/messages.ts` and `codec.ts`.
- **`rxfy-server` (new package)** — the server side of live updates:
  - `defineResource` derives an rxfy model, Zod schema, and key extractor from a Drizzle table (no codegen), and optionally accepts a pre-made `model` so a live resource can share a `ModelDescriptor` with client code (`packages/rxfy-server/src/resource.ts`).
  - `createResourceRegistry` indexes resources by name (`resource-registry.ts`).
  - `createServer` write functions (`update`/`create`/`delete`/`touch`) persist via Drizzle and broadcast over a hub; `createInMemoryHub` provides the default pub/sub (`server.ts`, `hub.ts`).
  - `grant` mints hashed topic ids for a response's entities and state channels, backed by `createTopicKeyer` (windowed HMAC derivation) and `invalidationChannel` (window/partition-aware channel derivation) for capability-based live-update auth (`topic-key.ts`, `state-channel.ts`).
  - A browser-safe `rxfy-server/browser` subpath exports the resource API without the Node-only `node:crypto` topic keyer, so `defineResource` can be imported into client bundles (`browser.ts`).
- **`rxfy-ws` (new package)** — the default WebSocket transport: `createWsServer(hub)` bridges the rxfy-server hub to sockets, and `rxfy-ws/client`'s `createWsClient` is a cross-platform client transport with reconnect and subscription replay (`packages/rxfy-ws/src/server.ts`, `client.ts`).
- **`rxfy-react` client live layer** — `createLiveClient` applies inbound entity patches to stores and counts per-state staleness; new exports `stateChannel`, `readSsrGrants`, `LiveClientContext`/`useLiveClient`, and a `liveClient` prop on `StoreProvider`. `useStateData` now returns `updatesAvailable$` and `applyUpdates()` for opt-in refresh UX (`packages/rxfy-react/src/live/`, `useStateData.ts`).
- **`rxfy` live-aware core** — `defineState` accepts an optional `window` field naming the pagination/slice params excluded from a state's live invalidation channel (`packages/rxfy/src/state/state.ts`), and the SSR hydration payload carries live-update grants via `DehydratedState.grants` (`packages/rxfy/src/ssr/hydration.ts`).

### Breaking Changes

- **`QueryCache`: `getPromise` + `setPromise` replaced by `getOrStart(key, start)`** (`packages/rxfy/src/query/query-cache.ts`). The cache now owns the check-and-store atomically: `start` runs only on a cache miss and its promise is registered (and auto-cleared on settle) automatically, so callers can no longer create an in-flight fetch without deduping it. Migrate:

  ```ts
  // before
  const existing = cache.getPromise(key);
  if (existing) return existing;
  const promise = fetchQuery();
  cache.setPromise(key, promise);
  return promise;

  // after
  return cache.getOrStart(key, fetchQuery);
  ```

- **`QueryCache`: `peek` and `delete` removed.** Read a query's current value via `getQuery(key).get()` instead of `peek(key)`.

### Dependencies

- `rxfy-react` now depends on `rxfy-protocol` (workspace-versioned) for the live-update wire types.
- `rxfy-protocol` depends on `superjson` for rich-type serialization.
- `rxfy-server` depends on `rxfy-protocol`, with peer dependencies on `drizzle-orm` (^0.45.0), `drizzle-zod` (^0.8.0), `rxfy` (^2.0.0), and `zod` (^4.0.0).
- `rxfy-ws` depends on `rxfy-protocol`, with peer dependencies on `ws` (^8.0.0) and an optional `rxfy-server` peer for the server adapter.
