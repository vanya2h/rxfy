# Optimistic mutations

Update the UI the moment a user acts, send the request in the background, and reconcile
when it returns — rolling back if it fails. rxfy ships no optimistic helper; you apply the
change to the store yourself, then confirm or revert. This guide shows the two cases:
editing one entity, and changing what a list contains.

## Editing one entity

A toggle or rename changes a single todo's **value**. Snapshot it, apply the change, then
reconcile with the server's canonical row or restore the snapshot on failure. Work at the
store level — [`store.getValue`](/models-state#low-level-storage) reads the current value
synchronously and `store.set` writes it back:

```tsx
import { useModelStore } from "rxfy-react";
import { TodoModel } from "./models";
import { api } from "./api";

function useToggleTodo() {
  const store = useModelStore(TodoModel);

  return async function toggle(id: string) {
    const before = store.getValue(id);
    if (!before) return;

    store.set(id, { ...before, done: !before.done }); // optimistic

    try {
      const saved = await api.toggleTodo(id); // server returns the canonical row
      store.set(id, saved); // reconcile — server wins
    } catch {
      store.set(id, before); // rollback to the snapshot
    }
  };
}
```

Every component subscribed to this todo reflects each `store.set` immediately — the
optimistic value, then the reconciled one (or the restored snapshot). No re-fetch, no
spinner.

## Changing list membership

Adding or removing a row changes which **ids** a query lists, not just an entity value.
Drive it through the [`useStateData`](/react#usestatedata) handle: a mutation updates the
id list (and the entity cells under it) right away, and `reload()` re-pulls the canonical
list if the request fails.

```tsx
import { useStateData } from "rxfy-react";
import { todosState, fetchTodos } from "./todos";
import { api } from "./api";

function useAddTodo(params: { filter: "all" | "active" | "done" }) {
  const { mutations, reload } = useStateData(todosState, fetchTodos, params);

  return async function add(title: string) {
    const todo = { id: crypto.randomUUID(), title, done: false };
    mutations.addTodo(todo); // optimistic insert into the id list + store

    try {
      await api.createTodo(todo);
    } catch {
      reload(); // rollback: re-pull the authoritative list
    }
  };
}
```

`reload()` deletes the query's cache entry and fetches again, so a failed insert disappears
on the next render. When you need the rollback to be instant (no round-trip), snapshot the
id list before the mutation and `set` it back instead:

```tsx
const { data$, set, mutations } = useStateData(todosState, fetchTodos, params);

// inside the handler, before mutating:
const before = await firstValueFrom(data$); // { todos: string[] }
mutations.addTodo(todo);
// on failure:
set((prev) => ({ ...prev, todos: before.todos }));
```

## Reconciling with server-assigned ids

When the server, not the client, assigns the id, the optimistic row carries a temporary id
that the real row won't match. Insert with a temp id, then on success replace it: remove
the temp entity and `setMany` the saved one, or `reload()` to let the canonical list arrive.
For a client-generated id (`crypto.randomUUID()` above) there is nothing to reconcile — the
optimistic and saved rows share an id, so the second `set` lands on the same cell.

## Choosing a strategy

- **Per-entity edits** (toggle, rename) → snapshot the entity, `store.set`, reconcile or
  restore. Cheapest and most precise.
- **Membership changes** (add, delete) → mutate through the handle, `reload()` to roll back,
  or snapshot the id list for an instant revert.
- When in doubt, `reload()` is the simplest rollback: the server is the source of truth, and
  a re-fetch can never leave the list in a wrong state.
