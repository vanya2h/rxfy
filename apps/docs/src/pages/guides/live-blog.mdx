# Live blog [Build a real-time blog: SSR, then live patches and stale badges]

This tutorial walks through `examples/vite-blog-framework` in the order you would build it: define
the schema and resources, stand up a live server, wire WebSockets, write through the API, SSR with
grants, and finally hydrate on the client. By the end you have a blog where edits propagate to every
open tab in real time and new or deleted posts show a "click to refresh" badge.

:::tip
Follow along in the actual example ŌĆö run it, open two tabs, and watch the behavior described in
each step before moving on to the next.
:::

## What you'll build

A server-rendered blog with three entity types ŌĆö users, posts, comments ŌĆö and two live behaviors:

- **Patch** ŌĆö editing a post or comment body pushes the change silently to every connected tab;
  no spinner, no reload.
- **Stale** ŌĆö creating or deleting a post or comment marks the affected view as stale; a
  "click to refresh" badge appears rather than reordering or removing items from under the reader.

**Stack:** Vite SSR ┬Ę Hono ┬Ę `@hono/node-ws` ┬Ę PGlite ┬Ę Drizzle ORM ┬Ę `rxfy-server` ┬Ę `rxfy-ws` ┬Ę `rxfy-client`

### Define resources

Resources are the bridge between your Drizzle tables and rxfy's normalized model stores.
[rxfy-server](/framework/server) exports `defineResource` and `createResourceRegistry`:

```ts [src/blog/resources.ts]
import { commentModel, postModel, userModel } from "./models.js";
import { createResourceRegistry, defineResource } from "rxfy-server/browser";
import { comments, posts, users } from "../db/schema.js";

export const userResource = defineResource({ table: users, model: userModel });
export const postResource = defineResource({ table: posts, model: postModel });
export const commentResource = defineResource({ table: comments, model: commentModel });

export { commentModel, postModel, userModel };

export const resources = createResourceRegistry([userResource, postResource, commentResource]);
```

`defineResource` pairs a Drizzle table with the rxfy model that governs it ŌĆö the model's `getKey`
tells the live server which column is the primary key, and its schema validates incoming rows.
`createResourceRegistry` collects all three so the server has a single object to route live events
through.

The Drizzle schema itself is ordinary Drizzle ŌĆö three `pgTable` definitions (users, posts,
comments) with text primary keys and timestamp defaults. Nothing rxfy-specific is required there.

:::warning[Entity ids must be unguessable]
Entity patches carry full rows and fan out on raw `name:id` topics gated only by a valid grant.
With guessable ids (serial integer PKs) any valid grant lets a client watch arbitrary rows ŌĆö so
ids of live models must be opaque. Use UUIDs (`crypto.randomUUID()`) for every id these tables
generate.
:::

### Stand up the live server

The live server owns the database connection and the in-memory pub/sub hub, which keys
subscriptions by **socket**. Create it once and export it:

```ts [server/live.ts]
import { createInMemoryHub, createServer } from "rxfy-server";
import { resources } from "../src/blog/resources.js";
import { db } from "./db.js";

// The hub holds live socket subscriptions, so there must be exactly ONE instance. entry-server
// receives `live` as a parameter instead of importing this module, so a separate Vite SSR module
// graph never instantiates a second hub.
export const hub = createInMemoryHub();
export const SECRET = process.env.RXFY_SECRET ?? "dev-secret-change-me";

export const live = createServer({ db, resources, hub, secret: SECRET });
```

- `createInMemoryHub` ŌĆö pub/sub bus for the current process, keyed by socket. Replace with a
  Redis adapter for multi-instance deployments.
- `secret` ŌĆö the HMAC key the server signs grants with. Read it from `process.env.RXFY_SECRET`
  and share the same value with the WebSocket server, which verifies grants against it.
- There is no server-held session: the server signs a per-state grant on `live.serve` /
  `live.hydration`, and the client subscribes with it ŌĆö see
  [Grants](/framework/server/grants).
- `createServer` ŌĆö wires the hub, resource registry, and secret together into the `live` object
  your API routes will call.

### Wire the WebSocket endpoint

`createWsServer` from [rxfy-ws](/framework/ws) turns the hub into a WebSocket server. The Hono
`upgradeWebSocket` adapter bridges Hono's WS lifecycle to rxfy's connection interface:

```ts [server/ws.ts]
import { EventEmitter } from "node:events";
import type { UpgradeWebSocket } from "hono/ws";
import { createWsServer } from "rxfy-ws";
import { hub, SECRET } from "./live.js";

const wsServer = createWsServer(hub, { secret: SECRET });

export function liveRoute(upgradeWebSocket: UpgradeWebSocket) {
  return upgradeWebSocket(() => {
    const emitter = new EventEmitter();
    return {
      onOpen(_evt: Event, ws: { send: (data: string) => void }) {
        wsServer.handleConnection({
          send: (data: string) => ws.send(data),
          on: (event, cb) => emitter.on(event, cb),
        });
      },
      onMessage(evt: MessageEvent) {
        emitter.emit("message", evt.data);
      },
      onClose() {
        emitter.emit("close");
      },
    };
  });
}
```

`createWsServer(hub, { secret })` returns a handler that speaks the rxfy subscription protocol and
verifies each `subscribe` frame's grant with that shared secret. The `EventEmitter`
adapts Hono's callback API to the `on("message")` / `on("close")` interface rxfy expects. Mount
`liveRoute(upgradeWebSocket)` on a `/live` path in your Hono app.

### Write through the server

All mutations go through `live.create`, `live.update`, and `live.delete`. Each method writes to the
database **and** broadcasts the appropriate live event to subscribed clients.

:::code-group

```ts [server/api.ts (setup)]
import { touch } from "rxfy-server";
import { commentResource, postResource } from "../src/blog/resources.js";
import { postDetailState, postsState } from "../src/blog/states.js";
import { live } from "./live.js";

// A state doubles as its channel descriptor: touch(postsState, params) marks it stale.
```

```ts [server/api.ts (posts)]
// POST /posts ŌĆö create a post, mark the posts list as stale
const row = await live.create(
  postResource,
  { id: newId(), userId, title, body },
  { touch: [touch(postsState, {})] },
);

// PATCH /posts/:id ŌĆö update fields, push a live patch to every reader
const row = await live.update(postResource, c.req.param("id"), patch);

// DELETE /posts/:id ŌĆö delete the row, mark the posts list as stale
await live.delete(postResource, c.req.param("id"), { touch: [touch(postsState, {})] });
```

```ts [server/api.ts (comments)]
// POST /posts/:id/comments ŌĆö create a comment, stale the post detail view
const row = await live.create(
  commentResource,
  { id: newId(), postId, name, body },
  { touch: [touch(postDetailState, { postId })] },
);

// PATCH /comments/:id ŌĆö update the comment body, push a patch
const row = await live.update(commentResource, c.req.param("id"), patch);

// DELETE /posts/:postId/comments/:id ŌĆö delete, stale the post detail view
await live.delete(commentResource, c.req.param("id"), { touch: [touch(postDetailState, { postId })] });
```

:::

**`update` emits a `patch`** ŌĆö the new field values are merged into the normalized store on every
connected client immediately, with no round-trip. The UI re-renders in place.

**`create` and `delete` with `touch` emit a `stale`** ŌĆö the named state channel (e.g.
`postsState`'s) is flagged as stale on every client that holds it. The client does not re-fetch
automatically; instead it surfaces a badge so the user can refresh when convenient.

`touch(state, params)` identifies the exact query to stale. For the posts list that is
`touch(postsState, {})` (no params); for a post detail it is
`touch(postDetailState, { postId })` so only readers of that specific post see the badge.

:::note
`live.update` with no `touch` option produces only a `patch`. Only mutations that change the
membership of a list (create, delete) need `touch` to signal the list is outdated.
:::

### One API client for both environments

The client never imports server code ŌĆö the hono endpoints are the single source of truth, and SSR
and the browser consume them through the same typed RPC client. `createApiClient` builds it in one
of two modes:

```tsx [src/blog/api-client.tsx (client)]
import type { Hono } from "hono";
import { hc } from "hono/client";
import type { AppType } from "../../server/api.js";

/** The shape of hono's in-process `app.request` ŌĆö what the server entry injects for SSR. */
export type ApiFetch = Hono["request"];

export function createApiClient(serverFetch?: ApiFetch) {
  return serverFetch
    ? hc<AppType>("http://ssr.internal", { fetch: serverFetch })
    : hc<AppType>("/api");
}
```

- **Browser** ŌĆö no argument: the client targets `/api` over the network, plain. Live
  subscriptions ride the `$grant` in each response payload, so the fetch client carries nothing.
- **SSR** ŌĆö the server entry passes hono's in-process `app.request`, so requests to the
  `http://ssr.internal` base are answered by the same routes without ever touching the network ŌĆö
  no duplicated queries, no server imports in client code.

The same module exports `ApiProvider` ŌĆö a plain React context provider for the client ŌĆö and
`useApi()`, which returns the typed client from context. There is no wrapper layer: components call
the endpoints directly, whether as `useStateData`'s `fetchFn` or for writes:

```tsx [src/App.tsx (abbreviated)]
// Each route component owns its data: useStateData + an inline fetchFn over the typed client.
// The view components it renders are handle-driven ŌĆö they know nothing about fetching.
function HomeRoute() {
  const api = useApi();
  const posts = useStateData({
    state: postsState,
    fetchFn: async () => (await api.posts.$get()).json(),
    params: {},
  });
  return <PostList posts={posts} />;
}

function PostRoute({ postId }: { postId: PostId }) {
  const api = useApi();
  const detail = useStateData({
    state: postDetailState,
    fetchFn: async ({ postId }) => {
      const res = await api.posts[":id"].$get({ param: { id: postId } });
      if (!res.ok) throw new Error(`Post ${postId} not found`);
      return res.json();
    },
    params: { postId },
  });
  return <PostDetail detail={detail} />;
}

// writes ŌĆö call the endpoint directly from any component with useApi()
await api.posts[":id"].comments.$post({ param: { id: postId }, json: input });
```

`useStateData` intentionally ignores `fetchFn` identity, so inline async arrows like these never
retrigger a fetch ŌĆö and the result type is inferred from the endpoint (a union for routes that
declare error variants). Data acquisition stays in the route components; the views receive the
resulting state handle as a prop and only render it.

### SSR with grants

During server-side rendering the server populates a `ModelRegistry` with the fetched entities, and
`useStateData` logs each rendered state's channel into `registry.channels`. One call at the end of
the render signs one grant per rendered channel, embeds them alongside the dehydrated registry, and
returns the hydration script:

```tsx [src/entry-server.tsx]
import { PassThrough } from "node:stream";
import { StrictMode, Suspense } from "react";
import { renderToPipeableStream } from "react-dom/server";
import { createModelRegistry } from "rxfy";
import { StoreProvider } from "rxfy-react";
import type { RenderFn } from "../server/render-types.js";
import { ApiProvider, createApiClient } from "./blog/api-client.js";
import { App } from "./App.js";

export const render: RenderFn = (url, live, apiFetch) => {
  // SSR data fetching goes through the server's own endpoints, in-process.
  const apiClient = createApiClient(apiFetch);
  const registry = createModelRegistry();

  return new Promise((resolve, reject) => {
    const { pipe } = renderToPipeableStream(
      <StrictMode>
        <StoreProvider registry={registry} ssr>
          <ApiProvider client={apiClient}>
            <Suspense fallback={null}>
              <App url={url} />
            </Suspense>
          </ApiProvider>
        </StoreProvider>
      </StrictMode>,
      {
        onAllReady() {
          const sink = new PassThrough();
          let html = "";
          sink.on("data", (chunk: Buffer) => (html += chunk.toString()));
          sink.on("end", () => {
            // hydration() signs one grant per rendered channel and embeds them alongside the
            // dehydrated registry ŌĆö the client lifts the grants on hydration and subscribes.
            resolve({ html, state: live.hydration(registry) });
          });
          pipe(sink);
        },
        onError: (error) => reject(error instanceof Error ? error : new Error(String(error))),
      },
    );
  });
};
```

`render` takes `live` and `apiFetch` as parameters rather than importing them ŌĆö in dev, Vite's SSR
module graph is separate from the server's, and the hub (which holds subscription state), the db,
and the api must stay single instances across both. The contract is the shared `RenderFn` type in
`server/render-types.ts`; the server side passes hono's in-process `api.request` directly (it is a
bound class-field arrow, safe to detach), and the entry builds `createApiClient(apiFetch)` from it,
so SSR fetches hit the same endpoints without a network trip:

```ts [server/render.ts (abbreviated)]
// SSR data fetching goes through the same hono endpoints as the browser, in-process ŌĆö the render
// must use THIS module graph's api (and its db/live), not a Vite-side copy.
// ...load `render` from Vite in dev or from dist/server in production, then:
const rendered = await render(url, live, api.request);
```

See [Grants](/framework/server/grants) for the full model.

A plain API read endpoint signs the grant for its own response the same way, per request, with
`live.serve` ŌĆö which attaches it as `$grant` on the returned payload:

```ts [server/api.ts (serve per route)]
.get("/posts", async (c) => {
  const rows = await db.select().from(posts);
  return c.json(live.serve(postsState, {}, { posts: rows }));
})
```

### Hydrate and go live

On the client, create the WebSocket transport and live client ŌĆö the SSR-signed grants are lifted
automatically ŌĆö then pass the live client to `StoreProvider`:

```tsx [src/entry-client.tsx]
import { StrictMode } from "react";
import { hydrateRoot } from "react-dom/client";
import { createModelRegistry } from "rxfy";
import { createLiveClient, StoreProvider } from "rxfy-react";
import { createWsClient } from "rxfy-ws/client";
import { ApiProvider, createApiClient } from "./blog/api-client.js";
import { App } from "./App.js";

const registry = createModelRegistry();
const apiClient = createApiClient(); // browser client: plain network trip
const liveClient = createLiveClient({
  registry,
  transport: createWsClient({ url: `${location.protocol === "https:" ? "wss" : "ws"}://${location.host}/live` }),
  renewUrl: "/api/live/renew",
});

hydrateRoot(
  document.getElementById("root") as HTMLElement,
  <StrictMode>
    <StoreProvider registry={registry} ssr liveClient={liveClient}>
      <ApiProvider client={apiClient}>
        <App url={location.pathname} />
      </ApiProvider>
    </StoreProvider>
  </StrictMode>,
);
```

- `createWsClient` ŌĆö builds the WebSocket transport. The URL switches between `ws://` and `wss://`
  based on the page protocol.
- `createLiveClient` ŌĆö lifts each state's `$grant` (and the SSR grants via `readSsrGrants`),
  subscribes on the socket with the grant plus the entity topics its payload normalized into,
  renews grants before expiry against `renewUrl`, and replays the whole grant set on reconnect ŌĆö
  wiring incoming `patch` / `stale` events to the shared registry.
- `renewUrl: "/api/live/renew"` ŌĆö the app-mounted endpoint the client POSTs expiring grants to;
  it reissues each with `live.renew` behind the app's own auth.
- `ApiProvider client={apiClient}` ŌĆö hands the browser API client to every `useApi` call site,
  mirroring what the server entry did with the in-process client during SSR.
- `StoreProvider liveClient={liveClient}` ŌĆö makes the live client available to all hooks so that
  `useStateData` and `useModelStore` calls can receive stale notifications.

See [Live client](/react/live-client) for the full client API.

## See it live

1. Open the app in two browser tabs.
2. **Edit a post body in tab A.** Tab B silently updates ŌĆö no spinner, no reload. The body field
   was sent as a `patch` by `live.update` and merged directly into the normalized store.
3. **Create a new post in tab A.** Tab B shows a refresh badge on the posts list. The server called
   `touch(postsState, {})`, which emitted a `stale` event. The existing list is not disrupted;
   the reader chooses when to reload.
4. **Delete the post in tab A.** The same refresh badge appears in tab B for the same reason.

The distinction matters for user experience: patches are invisible and immediate; stale signals
let the user read without interruption and reload at will.

## What you built

- A **resource registry** that pairs Drizzle tables with rxfy models.
- A **live server** (`rxfy-server`) that writes to the database and broadcasts events through an
  in-memory hub.
- A **WebSocket endpoint** (`rxfy-ws`) bridged to Hono's WS adapter.
- API routes that distinguish between `patch` (edit) and `stale` (create/delete) live events by
  using `live.update` vs `live.create`/`live.delete` with `touch`.
- A **single typed API client** (`createApiClient` / `useApi`) over those routes, consumed
  in-process during SSR (hono's `app.request`) and over the network in the browser.
- **Channel grants** signed by `live.serve` and `live.hydration` and delivered in the payload, so
  the client subscribes for exactly what it was served.
- A **live client** on the browser that lifts grants, subscribes, renews, and applies patches in
  place while surfacing stale badges.
