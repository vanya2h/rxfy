# Pagination and infinite scroll

Load a list one page at a time and append each page into a single growing list, with every
row normalized like any other entity. [`useStateData`](/react#usestatedata) owns the first
page (and its cache / SSR); you fetch later pages yourself and merge them with `set`.

The cursor is **view state**, not entity data — it tells you where to resume, so it lives in
the component, outside the normalized store.

## Append pages with `set`

`set((prev) => …)` takes the full list of entities, runs your updater, and normalizes the
result back. Returning `prev.items` plus the next page appends those rows to the query's id
list and writes their entities to the store:

```tsx
import { useCallback, useMemo, useRef } from "react";
import { useStateData, Pending } from "rxfy-react";
import { feedState, FeedItemModel } from "./feed";
import { api } from "./api";

function Feed() {
  const params = useMemo(() => ({ filter: "all" as const }), []);
  const cursor = useRef<string | null>(null);

  // First page goes through useStateData; stash the next cursor as it loads.
  const fetchFirst = useCallback(async (p: { filter: "all" }) => {
    const page = await api.feed({ ...p, cursor: null });
    cursor.current = page.nextCursor;
    return { items: page.items };
  }, []);

  const { data$, set } = useStateData(feedState, fetchFirst, params);

  const loadMore = useCallback(async () => {
    if (!cursor.current) return; // no more pages
    const page = await api.feed({ cursor: cursor.current });
    cursor.current = page.nextCursor;
    set((prev) => ({ items: [...prev.items, ...page.items] }));
  }, [set]);

  return (
    <Pending value$={data$} pending={<p>Loading…</p>}>
      {({ items }) => (
        <>
          <ul>{items.map((id) => <FeedRow key={id} id={id} />)}</ul>
          <button onClick={loadMore}>Load more</button>
        </>
      )}
    </Pending>
  );
}
```

`data$` emits `{ items: string[] }` — ids only — so each page's new ids append to the list
while the row data lives in the `FeedItemModel` store. A row that appears on two pages (a
duplicate the server returned) resolves to one cell, not two.

## Drive it from a scroll sentinel

Swap the button for an `IntersectionObserver` to load the next page as the user reaches the
bottom:

```tsx
import { useEffect, useRef } from "react";

function LoadMoreSentinel({ onVisible }: { onVisible: () => void }) {
  const ref = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const el = ref.current;
    if (!el) return;
    const io = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) onVisible();
    });
    io.observe(el);
    return () => io.disconnect();
  }, [onVisible]);

  return <div ref={ref} />;
}
```

Render `<LoadMoreSentinel onVisible={loadMore} />` after the list. Guard against overlapping
loads with an `isLoading` ref if a fast scroll can fire `loadMore` twice before a page
returns.

## Notes

- **Keep the cursor out of `params`.** If the cursor were a `useStateData` param, each page
  would be a separate cached query and replace the previous list instead of extending it.
  Stable params plus manual `set` is what accumulates one list.
- **A new filter resets the list.** Changing `params` (a different `filter`) starts a fresh
  query — page one of the new filter. Reset `cursor.current` to `null` when the filter
  changes so the next `loadMore` starts over.
- **Reloading.** `reload()` re-fetches the first page only. To restore a deep scroll
  position after a reload, re-run your `loadMore` calls, or page the data server-side by an
  offset you persist.
