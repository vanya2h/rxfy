# Why rxfy? [The thinking behind the design]

Every data-rich app faces the same coordination burden: keep entities consistent when they change, invalidate caches after mutations, rehydrate server state on the client. Most projects wire this up from scratch.

## The problem

You have a list of todos. You update one: mark it done, rename it from a detail view, or get a websocket push that the server already made the change.

Most apps reach for one of three options.

- **Refetch the list.** You send a network round-trip to retrieve data you already have. On large lists or slow networks, the latency shows.
- **Patch in place.** You update the item in the list array. It works until a second component renders the same entity from a different query. Now you have two copies that can diverge.
- **Coordinate the cache manually.** Invalidate queries, call `setQueryData`, tune `staleTime`. You're maintaining a dependency graph between cached queries by hand. Miss one, and the first symptom is a stale UI in production.

SSR compounds it. You server-render the page with data, ship the HTML, then need to rehydrate so the client skips what it already fetched. Each library handles this differently, and most leave the wiring to you.

## The insight

All three options are workarounds for the same thing: your app holds multiple copies of the same entity. The list has one copy. The detail component fetches another. The cache might have a third. Every write has to find and update all of them.

Remove the copies and the workarounds become unnecessary.

Put every entity in one place, keyed by its id. Components don't hold data; they subscribe to an id. When something writes to that slot, every subscriber sees the update. The list doesn't re-fetch; it was never holding the data.

The same model works for SSR. The server fills the slots and serializes them. The client restores the snapshot and skips what it already has.

## The principles

- **One entity, one place.** `createModel` declares the Zod schema and an id extractor. Every entity lands in one `ModelStore` slot. `store.set` is the only write path; you can't fork a copy.
- **Streams as delivery.** Each store slot is an RxJS `Observable`. Components subscribe to the entity they need. A write notifies that slot's subscribers and nothing else re-runs.
- **Declaration over wiring.** `defineState` bundles the fetch params, model shape, mutation logic, and SSR snapshot into one declaration. You describe what the state is; rxfy handles the rest.
