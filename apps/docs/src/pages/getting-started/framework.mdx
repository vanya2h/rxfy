# Framework quickstart [The full stack for a live app]

The framework path puts the same normalized store behind a persistent connection. The
server owns writes: it applies changes in database, then publishes the new entity value automatically; every connected
client writes that value into its shared store, and every component subscribed to that
entity re-renders. Same store, same components — nothing in `useStateData` or
`useModelStore` is live-aware — the write just arrives over the wire instead of from a
local call.

This page walks the whole loop: install, wire the server, write through it, and watch an
edit land in another browser tab.

:::note[Is this path for you?]
The framework fits best when you're starting a new app, or when your backend already runs
on Node with [Drizzle](https://orm.drizzle.team) — `rxfy-server` binds live resources
directly to Drizzle tables. With a different backend stack, stay on the
[Store quickstart](/getting-started/store) path and fetch your API your way; you can
still add [SSR](/core-concepts/ssr) without the live stack.
:::

## Scaffold a new app

The fastest start is the official scaffolder. It creates a standalone, fully server-rendered
live app — Vite SSR, React Router, a Hono server, Drizzle on embedded Postgres
([PGlite](https://pglite.dev)), and real-time todos wired end to end:

:::code-group

```bash [npm]
npm create rxfy-app@latest my-app
```

```bash [pnpm]
pnpm create rxfy-app my-app
```

```bash [yarn]
yarn create rxfy-app my-app
```

:::

Then:

```bash
cd my-app
pnpm install
pnpm dev
```

Open http://localhost:3000 in two tabs and toggle a todo — the other tab updates instantly.
Everything below explains what the scaffold wired up; read on to understand it or to add
the framework to an existing app by hand.

## Install

Everything from the store path plus the live stack: `rxfy-server` (writes and
publishing) and `rxfy-ws` (the WebSocket transport), with
[Drizzle](https://orm.drizzle.team) and `ws` as their peers.
The wire contract between them (see [Live messages](/framework/server/messages)) comes
along as an internal dependency.

:::code-group

```bash [npm]
npm install rxfy rxfy-react rxfy-server rxfy-ws
npm install rxjs zod lodash drizzle-orm drizzle-zod ws
```

```bash [pnpm]
pnpm add rxfy rxfy-react rxfy-server rxfy-ws
pnpm add rxjs zod lodash drizzle-orm drizzle-zod ws
```

```bash [yarn]
yarn add rxfy rxfy-react rxfy-server rxfy-ws
yarn add rxjs zod lodash drizzle-orm drizzle-zod ws
```

```bash [bun]
bun add rxfy rxfy-react rxfy-server rxfy-ws
bun add rxjs zod lodash drizzle-orm drizzle-zod ws
```

:::

## Bind tables to models

A resource pairs a Drizzle table with the rxfy model that governs it — the model's name
becomes the live topic namespace, and its `getKey` locates the primary key. The registry
gives the server one object to route events through:

```ts [src/resources.ts]
import { createResourceRegistry, defineResource } from "rxfy-server/browser";
import { postModel } from "./models.js";
import { posts } from "./db/schema.js";

export const postResource = defineResource({ table: posts, model: postModel });
export const resources = createResourceRegistry([postResource]);
```

## Stand up the live server

`createServer` wires the database, the resources, and a pub/sub hub into the `live`
object your API routes will call:

```ts [server/live.ts]
import { createInMemoryHub, createServer } from "rxfy-server";
import { resources } from "../src/resources.js";
import { db } from "./db.js";

export const hub = createInMemoryHub();

export const live = createServer({ db, resources, hub });
```

- `createInMemoryHub` — the pub/sub backbone for one process, keyed by **session**; swap it
  for a distributed adapter when you scale out.
- There is no keyer, no grants, and no client-issued subscriptions. The server tracks what
  each browser session was served (via `live.serve` / `live.hydration`) and pushes updates
  for exactly that — see [Sessions](/framework/server/sessions).

## Serve the socket

`createWsServer(hub)` speaks the subscription protocol; a plain `ws` socket satisfies its
connection interface directly:

```ts [server/ws.ts]
import { WebSocketServer } from "ws";
import { createWsServer } from "rxfy-ws";
import { hub } from "./live.js";

const wsServer = createWsServer(hub);
// `httpServer` is the node:http server your app already listens on
const wss = new WebSocketServer({ server: httpServer, path: "/live" });

wss.on("connection", (socket) => wsServer.handleConnection(socket));
```

Framework servers that wrap the raw socket (Hono, Bun) need a thin adapter — see
[rxfy-ws](/framework/ws) for the Hono bridge.

## Write through the server

All mutations go through `live.create`, `live.update`, and `live.delete`. Each runs the
SQL write, then publishes:

```ts [server/api.ts]
import { touch } from "rxfy-server";
import { live } from "./live.js";

// edit — publishes a `patch`; every subscribed client applies it in place
await live.update(postResource, postId, { title, body });

// create / delete — publish `stale` on the state channels that list this entity
await live.create(postResource, { id: newId(), title, body }, { touch: [touch(postsChannel, {})] });
await live.delete(postResource, postId, { touch: [touch(postsChannel, {})] });
```

## Go live on the client

Give `StoreProvider` a live client built from the WebSocket transport and the session the
server minted during SSR. Your components stay exactly as they were on the store path:

```tsx [src/entry-client.tsx]
import { createModelRegistry } from "rxfy";
import { createLiveClient, readSsrSession, StoreProvider } from "rxfy-react";
import { createWsClient } from "rxfy-ws/client";

const registry = createModelRegistry();
const liveClient = createLiveClient({
  registry,
  transport: createWsClient({ url: `ws://${location.host}/live` }),
  session: readSsrSession() ?? crypto.randomUUID(),
});

hydrateRoot(
  document.getElementById("root")!,
  <StoreProvider registry={registry} ssr liveClient={liveClient}>
    <App />
  </StoreProvider>,
);
```

There is no allow-list to read: the server already knows what this session was served — the
SSR render minted the session with `live.hydration(registry)` and embedded it in the page;
`readSsrSession()` adopts it, or the client mints its own id on a CSR-only load. See
[Sessions](/framework/server/sessions).

## Open two tabs

Edit a post in tab A. Tab B updates in place — no spinner, no reload, no code in the
component asking for it.

What happened, in order:

1. `live.update` ran the update to database and got the new row.
2. It published a `patch` message on the entity's topic (`post:42`) into the hub.
3. The hub forwarded it to every session that was served that entity — sessions the
   server tracked, not topics the client subscribed to.
4. Each client wrote the entity into its shared `ModelStore` — the same write as
   `store.set` on the store path.
5. Every component subscribed to that id re-rendered. The query's id list was untouched,
   and nothing is re-fetched and nothing is re-rendered except the updated entity.

Edits travel as `patch`; creates and deletes instead mark the lists that reference the
entity as `stale`, surfacing a "click to refresh" badge. The two message kinds — and why
lists aren't updated in place — are covered in [Live messages](/framework/server/messages).
