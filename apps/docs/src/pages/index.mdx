# rxfy [Typed, normalized, reactive state — built on RxJS]

**rxfy** (/ɑɹ ɪks faɪ/) is a minimalistic framework that helps you declare models, states, and normalized stores as an Observables-based data-flow layer in your React apps. Focused on consistency and granular reactivity at no extra cost.

rxfy is built on three principles:

- Every entity lives in a normalized store, accessed granularly by its id; an update reaches every subscriber automatically.
- Each page has its own state, built from data in the normalized stores; components are the granular consumers of that state — each updates only when the data it reads changes.
- States and stores are serializable: rxfy has first-class Server-Side Rendering (SSR) support.

[Why rxfy?](/why) explains the thinking behind this design.

## Quick taste

::::steps

### Define the model

A model binds a Zod schema to an id extractor. Every entity of this type lives in exactly one store slot, accessible via `getKey` method.

```tsx
import { z } from "zod";
import { createModel } from "rxfy";

const TodoModel = createModel({
  schema: z.object({ id: z.string(), title: z.string(), done: z.boolean() }),
  getKey: (x) => x.id,
  name: "todo",
});
```

### Define state and mutations

`defineState` declares the state and the way how it is fetched via provided fetch function. It supports mutations that changes the state. rxfy normalizes the result into model stores automatically.

```tsx
import { defineState, array } from "rxfy";

const todosState = defineState({
  key: "todos",
  params: z.object({
    filter: z.enum(["all", "active", "done"]),
  }),
  model: {
    todos: array(TodoModel),
    isOpen: z.boolean(),
  },
  mutations: {
    addTodo: (
      prev,
      todo: {
        id: string;
        title: string;
        done: boolean;
      },
    ) => ({
      ...prev,
      todos: [...prev.todos, todo],
    }),
  },
});
```

### Fetch and render the list

`useStateData` wires the state to your fetch function and returns a reactive `data$` stream. `Pending` handles the loading and error states declaratively. The query shape carries only ids — not full entities.

```tsx
import { Pending, useStateData } from "rxfy-react";

function TodoApp() {
  const { data$, mutations } = useStateData({
    state: todosState,
    fetchFn: fetchTodos,
    params: { filter: "all" },
  });

  return (
    <Pending value$={data$} pending={<p>Loading...</p>} rejected={({ error }) => <p>{String(error)}</p>}>
      {({ todos }) => (
        <ul>
          {todos.map((id) => (
            <TodoItem key={id} id={id} />
          ))}
        </ul>
      )}
    </Pending>
  );
}
```

### Subscribe to individual entities

Each item subscribes to its own store. A single `store.set` updates every component showing that entity, with no list re-fetch.

```tsx
import { useMemo } from "react";
import { useModelStore } from "rxfy-react";

function TodoItem({ id }: { id: string }) {
  const store = useModelStore(TodoModel);
  const todo$ = useMemo(() => store.get(id), [store, id]);

  return <Pending value$={todo$}>{(todo) => <li>{todo.title}</li>}</Pending>;
}
```

## How it compares

- **Redux Toolkit** — same normalized, immutable approach, but stream-first instead of selector-first, with fetching and SSR built in rather than wired up from RTK Query and slices.
- **MobX** — trades direct mutation for explicit immutable streams that serialize cleanly for SSR, and adds normalization and a query layer MobX leaves to you.
- **Jotai** — its atom is a config object resolved against a store; an rxfy `Atom` is a real RxJS Observable cell that carries its value and works outside React.
- **TanStack Query** — caches responses by query key and refetches to reconcile; rxfy normalizes into per-entity cells, so one write reaches every subscriber with no refetch.
