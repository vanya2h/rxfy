# rxfy [Typed, normalized, reactive state — built on RxJS]

**rxfy** (/ɑɹ ɪks faɪ/) is a reactive data-flow layer for your React app: declare typed models, states, and normalized stores, and scale from a client-only store to a fully live app with server-side rendering and real-time updates via websockets. It's built for consistency and granular RxJS-based reactivity at no extra cost.

:::code-group

```bash [npm]
npm install rxfy rxfy-react
```

```bash [pnpm]
pnpm add rxfy rxfy-react
```

```bash [yarn]
yarn add rxfy rxfy-react
```

:::

## The problem it solves

Keeping every view of your data in agreement is hard. Doing it across many connected clients,
in real time, is way harder.

Rename one todo from its detail view. The list, the sidebar counter, the search results: every
one of them now has to show the new title. Most apps refetch the list (a round-trip for data
you already have), patch the cache in place (two copies that drift apart), or invalidate caches
by hand (miss one and the UI goes stale). Now add a second client: Bob still sees the old title
that Alice just changed. SSR compounds it — after server-rendering the page, the rehydration
wiring is usually left to you.

## The insight

All three fixes are workarounds for the same root cause: your app holds multiple copies of the
same entity. The list has one copy, the detail view fetched another, the cache may hold a
third, and every write has to find and update all of them. Remove the copies and the
workarounds become unnecessary.

rxfy stores each entity **once**, in a normalized store keyed by its id. The state holds only
references to stored entities by id; components don't hold data — they subscribe to the exact
entities they render, so one write reaches every subscriber. The same model carries SSR: the
server fills the store and serializes it, and the client restores the snapshot and skips what
it already fetched. With websockets on top, the write crosses the network too: the server
persists it and publishes it to every connected client, and each client's store updates every
place where the entity is rendered.

## Four principles

The insight above unfolds into the four principles rxfy is built on:

- **Every entity lives in one normalized store.** A model declares the entity's schema; every entity lands in a single slot keyed by id. Writing to the slot
  is the only write path — you can't fork a copy — so an update reaches every subscriber
  automatically.
- **Each page declares its own state over those stores.** A state composes the fetch params,
  state shape, and mutation logic into one declaration that references data in shared normalized stores.
  Components are the granular consumers — use id references from the store and unwrap it in your component. Each [`Pending`](/react/pending) component re-renders only when that entity changes, nothing else.
- **Values unwrap as late as possible.** The data from normalized store is wrapped in [`Atom`](/rxfy/create-atom), and only the leaf
  component that renders a value unwraps it. A write skips propagates to all subscribers automatically. The narrower the unwrap, the narrower the
  re-render — see [Late Unwrapping](/core-concepts/late-unwrapping).
- **States and stores are serializable.** The same declaration the client renders from, the
  server fills and serializes; the client restores the snapshot and skips what it already
  fetched on server. [SSR](/core-concepts/ssr) has first-class support and works well with frameworks
  like Next.js or React Router v7.

## RxJS as a reactivity engine

rxfy doesn't invent a reactivity system. It builds on [RxJS](https://rxjs.dev) and exposes that
foundation instead of hiding it: an [`Atom`](/rxfy/create-atom) **is** an `Observable` with
synchronous `get()` and `set()`, and entity cells, state `data$`, and
[lenses](/rxfy/create-lens) are all streams.
That single decision pays off everywhere:

- **The whole operator library works on your app state.** Derived state is a `pipe`: `map`,
  `combineLatest`, `debounceTime`, `switchMap`, and the rest of RxJS compose directly with
  entities and states. Combine and transform any kind of data streams.
- **Push-based, granular delivery.** A write to the store pushes the new value to exactly that
  cell's subscribers. No re-running selectors: delivery is the Observable contract itself.
- **Live data fits natively.** A websocket push is already a stream, so server updates flow
  into the same normalized cells without an adapter layer. The
  [live framework](/getting-started/framework) is built on exactly this.
- **Values can stay wrapped.** Because data travels as Observables, nothing unwraps until the
  leaf component that renders it subscribes — see
  [Late Unwrapping](/core-concepts/late-unwrapping).
