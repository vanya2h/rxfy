# Grants & live hydration [Hand off an SSR render to a live client]

A live client subscribes to entity topics and state invalidation channels over a persistent
connection. The server must decide which subscriptions are allowed before the client connects —
clients cannot self-issue subscriptions, so the server mints a signed allow-list (grants) during
the SSR render and embeds it in the page.

## The problem

Every entity topic (`posts:42`) and state channel (`posts:author=7`) is a capability. If clients
could subscribe freely, any user could watch data they are not permitted to see. Grants are
capability tokens minted server-side from the rendered registry — only topics and channels that
were actually fetched during the render are included, and the token is tied to a rotating key the
server controls.

## On the server

`live.grant(registry, spec)` mints the allow-list from a rendered registry. Call it after
`onAllReady`, once the registry is fully populated, and spread the returned `grants` object into
`hydrationScript` alongside the dehydrated state:

`live` comes from [`createServer`](/framework/server/create-server); the `resources.ts` and
`routes.ts` tabs show the `entities` and `states` you hand to `live.grant`:

:::code-group

```tsx [entry-server.tsx]
import { dehydrate, hydrationScript } from "rxfy";
import { live } from "../server/live.js";
import { commentResource, postResource, userResource } from "./blog/resources.js";
import { matchRoute, routeStates } from "./routes.js";

onAllReady() {
  // collect pipe to `html` string, then:
  const grants = live.grant(registry, {
    entities: [postResource, userResource, commentResource],
    states: routeStates(route),   // Array<{ state: StateChannelDescriptor; params: Record<string, unknown> }>
  });
  resolve({ html, state: hydrationScript({ ...dehydrate(registry), grants }) });
}
```

```ts [src/blog/resources.ts]
import { commentModel, postModel, userModel } from "./models.js";
import { createResourceRegistry, defineResource } from "rxfy-server/browser";
import { comments, posts, users } from "../db/schema.js";

export const userResource = defineResource({ table: users, model: userModel });
export const postResource = defineResource({ table: posts, model: postModel });
export const commentResource = defineResource({ table: comments, model: commentModel });

export const resources = createResourceRegistry([userResource, postResource, commentResource]);
```

```ts [src/routes.ts]
import type { StateChannelDescriptor } from "rxfy-server";
import { postDetailState, postsState } from "./blog/states.js";

export type Route = { name: "home" } | { name: "post"; postId: string } | { name: "not-found" };

/** Parse a pathname into a route. */
export function matchRoute(pathname: string): Route {
  if (pathname === "/") return { name: "home" };
  const m = /^\/posts\/([^/]+)\/?$/.exec(pathname);
  if (m) return { name: "post", postId: decodeURIComponent(m[1]!) };
  return { name: "not-found" };
}

/** The state instances a route renders — used to mint grant channels during SSR. */
export function routeStates(route: Route): Array<{ state: StateChannelDescriptor; params: Record<string, unknown> }> {
  if (route.name === "home") return [{ state: postsState as unknown as StateChannelDescriptor, params: {} }];
  if (route.name === "post")
    return [{ state: postDetailState as unknown as StateChannelDescriptor, params: { postId: route.postId } }];
  return [];
}
```

:::

- `entities` — the resources whose loaded rows should be grantable. Only ids actually present in
  the registry are included, keyed through
  [`createTopicKeyer`](/framework/server/create-topic-keyer).
- `states` — concrete state instances (`{ state: StateChannelDescriptor; params }`) whose
  invalidation channels the client may watch.

`grants` has shape `{ entities: Record<string, string>; channels: Record<string, string> }` —
a topic-or-channel → hashed-key lookup the client uses when opening subscriptions.

Outside SSR you can mint grants the same way and return them directly in the API response
(`return c.json({ data, grants })`); the client reads them instead of `readSsrGrants()`.

## On the client

Read the embedded grants with `readSsrGrants()` and pass them to `createLiveClient`. Hand the
live client to `StoreProvider` so every `useStateData` call can subscribe automatically once the
socket opens.

```tsx [entry-client.tsx]
import { createModelRegistry } from "rxfy";
import { createLiveClient, readSsrGrants, StoreProvider } from "rxfy-react";
import { createWsClient } from "rxfy-ws/client";

const registry = createModelRegistry();
const liveClient = createLiveClient({
  registry,
  transport: createWsClient({
    url: `${location.protocol === "https:" ? "wss" : "ws"}://${location.host}/live`,
  }),
  grants: readSsrGrants(),
});

hydrateRoot(
  document.getElementById("root")!,
  <StoreProvider registry={registry} ssr liveClient={liveClient}>
    <App url={location.pathname} />
  </StoreProvider>,
);
```

For the full `createLiveClient` API — reconnection, optimistic updates, and manual grant refresh
— see [Live client](/react/live-client).

## State channels

`invalidationChannel(state, params)` derives the channel name for a state instance. It is pure
and deterministic: the same inputs always produce the same string on both client and server.

Window dimensions (`page`, `cursor`, `sort`, …) listed in `state.window` are stripped before
building the channel, so every window of the same partition shares one channel. When the server
publishes a `stale` message on e.g. `posts:author=7`, every client subscribed to any page of
that author's post list receives the invalidation and knows to refetch.

```ts [state-channel.ts]
export type StateChannelDescriptor = {
  key: string;
  window?: readonly string[];   // params excluded from the channel (pagination dims)
};

// Pure, identical on client and server.
function invalidationChannel(
  state: StateChannelDescriptor,
  params: Record<string, unknown>,
): string;
```

:::info[No window keys in grants]
`states` passed to `live.grant` should use the same `params` object you passed to `useStateData`
— `invalidationChannel` strips the window keys internally, so the channel in the grant and the
channel the client derives from the same params always agree.
:::
