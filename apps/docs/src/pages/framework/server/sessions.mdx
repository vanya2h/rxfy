# Sessions [The server tracks what each browser was served, and pushes updates for it]

There are no grants and no client subscriptions. The server tracks what each browser session was
served and pushes updates for exactly that. The client's entire outbound protocol is one
`hello { session? }` frame — there is nothing else for it to say.

## Session identity

Minting is always the server's job — the client never invents an id. It happens on one of two
paths:

- **SSR loads** — [`live.hydration(registry)`](/framework/server/create-server) mints the session
  server-side and embeds it in the hydration payload; the client adopts it with `getSessionId()`.
- **Client-only loads** — the live client sends a session-less `hello` over the WebSocket; the
  server mints an id, answers with a `session` frame carrying it, and binds the socket. The live
  client adopts the id (and re-hellos so reconnects replay it). Until that assignment arrives,
  `getSessionId()` returns `undefined`.

`getSessionId` comes from `rxfy-client` (the framework-agnostic browser live runtime,
re-exported by `rxfy-react`). Once the id is known it is memoized — every caller sees the same
one.

Attach it to every API request; the live client picks it up by default. In the app templates the
wiring lives in `createApiClient` — the single typed client both environments consume (components
reach it through `ApiProvider` / `useApi()`); only the browser branch carries the header, since SSR
calls the routes in-process under the session the render itself mints:

```ts [src/api-client.tsx]
import { sessionHeaders } from "rxfy-client";

export function createApiClient(serverFetch?: ApiFetch) {
  return serverFetch
    ? hc<AppType>("http://ssr.internal", { fetch: serverFetch }) // SSR: hono's in-process app.request
    : hc<AppType>("/api", { headers: sessionHeaders }); // browser: network trip + session header
}

const liveClient = createLiveClient({ registry, transport: createWsClient({ url }) }); // session defaults to getSessionId()
```

`sessionHeaders()` returns `{ "x-rxfy-session": <id> }` for spreading into any HTTP client's
default headers; hono's `hc` also accepts it directly as a lazy headers function, as above. An
app with its own fetch layer wraps it instead: `withSession(myFetch)` (or `withSession()` for
the ambient `fetch`) returns a fetch that carries the session header once the session is known.
While it is unknown both return nothing extra — a headerless request is served normally, just
not recorded for live updates. In a client-only app, fetches that fire before the WebSocket
assignment arrives are therefore not recorded; the refresh / refetch path recovers.

## Serving = subscribing

A read endpoint wraps its result in `live.serve` — a pass-through that returns the data unchanged
and registers the served entities plus the state channel under the requester's session:

```ts [server/api.ts]
.get("/todos", async (c) => {
  const rows = await db.select().from(todos);
  return c.json(live.serve(c.req.raw, todosState, {}, { todos: rows }));
})
```

Pass the **same** `params` you pass to `useStateData` — window keys are stripped internally, so
the registered channel always matches the one the client's `updatesAvailable$` counts.

SSR renders register everything at once: `useStateData` logs each rendered state's channel into
`registry.channels` during SSR, and entities come from the render registry. One call at the end of
the render hands the whole thing to the client:

```tsx [entry-server.tsx]
onAllReady() {
  // collect pipe into `html`, then:
  resolve({ html, state: live.hydration(registry) });
}
```

The entry implements the shared `RenderFn` contract from `server/render-types.ts` — it receives
`live` and `apiFetch` (hono's in-process `app.request`) as parameters rather than importing them:
in dev, Vite's SSR module graph is separate from the server's, and the hub (which now holds
subscription state), the db, and the api must stay single instances across both.
`server/render.ts` calls `render(url, live, api.request)` — `request` is a bound class-field
arrow, safe to detach — and the entry builds `createApiClient(apiFetch)` from it, so SSR data
fetching goes through the server's own endpoints in-process.

## Lifecycle

- Pushes flow once the WebSocket `hello` binds the session; fetching before the socket connects is
  fine — the hub record waits.
- Reconnecting re-sends `hello` and delivery resumes. Updates published while disconnected are
  lost — the refresh badge / `applyUpdates` refetch is the recovery path, not replay.
- A session with no bound socket expires after the hub's `ttlMs` (default 5 minutes) — this covers
  an SSR render whose client hasn't opened the WebSocket yet, and a closed tab that might
  reconnect.

## What stays manual

- [`touch(channelDescriptor, params)`](/framework/server/writes) on writes — only the app knows
  which lists a write invalidates.
- One `live.serve` per read endpoint — the server can't see what a plain Drizzle read served.

See [`createServer`](/framework/server/create-server) for `live.serve` / `live.hydration` in the
full `Live` object, [`createInMemoryHub`](/framework/server/hub) for how the hub tracks sessions,
and [Live client](/react/live-client) for `createLiveClient` and the session helpers on the
browser side.
