# Examples

Runnable example apps in the [rxfy repository](https://github.com/vanya2h/rxfy/tree/main/examples).

## vite-todo

**[examples/vite-todo](https://github.com/vanya2h/rxfy/tree/main/examples/vite-todo)**

A classic Todo app built with Vite SSR. Covers the full rxfy loop: define a model, fetch with `useStateData`, render with `<Pending>`, subscribe to individual entities with `useModelStore`. SSR gives zero-flash first paint via `dehydrate` + `hydrationScript`.

```bash
pnpm --filter rxfy-example-todo-app dev
```

Companion guide: [Build a Todo app](/guides/todo-app)

## vite-realtime-todos

**[examples/vite-realtime-todos](https://github.com/vanya2h/rxfy/tree/main/examples/vite-realtime-todos)**

Normalized rxfy state driven by server-pushed updates over WebSockets, with targeted per-connection delivery. Built with Vite SSR · Hono · Drizzle (SQLite, in-memory).

The server tracks which entity ids each connection depends on and pushes only the changed entity — no broadcast, no refetch, no re-select. The client derives its subscription set straight from `registry.added$`, so the live layer stays in sync with exactly what the store has loaded.

```bash
pnpm --filter rxfy-example-realtime-todos dev
# open http://localhost:5175 in two tabs — toggle a todo in one, watch it update in the other
```

Companion guide: [Live updates over WebSockets](/guides/live-updates-websockets)

## next-blog

**[examples/next-blog](https://github.com/vanya2h/rxfy/tree/main/examples/next-blog)**

A blog app using Next.js App Router with streaming SSR. Demonstrates branded Zod types (`UserId`, `PostId`, `CommentId`), multiple related models (`User`, `Post`, `Comment`), and `<HydrationStream />` from `rxfy-react/next` for progressive hydration as Suspense boundaries resolve.

```bash
pnpm --filter rxfy-example-next-blog dev
```

## rr7-blog

**[examples/rr7-blog](https://github.com/vanya2h/rxfy/tree/main/examples/rr7-blog)**

The same blog domain as `next-blog`, but on React Router 7 (framework mode) with buffered SSR — a direct side-by-side of two SSR strategies. Here rxfy is the **single data layer**: a customized `entry.server` creates a per-request registry, renders with `renderToPipeableStream` + `onAllReady`, and injects `hydrationScript(dehydrate(registry))` before `</body>`; the client drains it for a zero-refetch first paint. React Router loaders are reserved for **routing only** — an index redirect and URL-param validation — never data fetching.

```bash
pnpm --filter rxfy-example-rr7-blog dev
```
