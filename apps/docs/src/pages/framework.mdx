# Framework (Real-time) [Publish live entity updates to every connected client]

rxfy ships a set of packages that wire server writes into client stores over a persistent
connection. When a value changes on the server, every component subscribed to that entity
re-renders — no polling, no manual refetch.

## When you need this

Use the framework packages when you want changes to appear live across clients as they
happen. If you only need normalized client state — fetch once, read reactively — you do
NOT need these packages. Stop at [Model Stores](/getting-started/store) instead.

## The three packages

- [`rxfy-server`](/framework/server) — write through the server and publish live updates.
- [`rxfy-protocol`](/framework/protocol) — the wire contract between server and client.
- [`rxfy-ws`](/framework/ws) — the default WebSocket transport.

## How it works underneath

Normalization does most of the work. Because every entity lives in one keyed cell (see
[Normalization](/core-concepts/normalization)), a server push doesn't need to know which
client renders what. The push carries a new entity value, the client writes it into the
shared store, and every component subscribed to that entity re-renders — no refetch, no
re-selection.

Normalization also defines the subscription set: the ids a client fetched and normalized
are the entities it should receive updates for. Each connection subscribes to the ids it
holds, and the server pushes a change only to the connections that hold that id. Your
existing `useStateData` / `useModelStore` components don't change; nothing in them is
live-aware.

## patch vs stale

There are two server→client message kinds:

**patch** — updates an entity's value (an edit). It reaches every connection subscribed to
that entity id and applies silently in place: the entity cell updates, subscribers
re-render, and the query's id list is untouched. `rxfy-server` publishes a patch
automatically when you call `live.update`.

**stale** — signals that a list or state channel has changed (a row was created or
deleted, so an id must appear in or vanish from a query's id array). Because that's a
change to the query shape rather than an entity value, the framework surfaces a refresh
affordance — a "N new — click to refresh" badge — rather than mutating the list in
place. `rxfy-server` publishes stale on the state channels you name via `touch()` in the
`opts` of `live.create` / `live.delete`.

:::info[Why lists aren't updated in place]
Inserting or removing an id from a live list would require the server to know the full
ordering and filter state of every client's query. `stale` avoids that coupling: the
client decides when and whether to reload, and the refetch uses the same `fetchFn` it
already has.
:::

## Data flow

```
Drizzle table → defineResource → model → live.update → hub.publish(patch) → WebSocket → client store → subscribers re-render
```

On `live.create` / `live.delete` with a `touch()` target, the hub additionally publishes a
`stale` message on the named state channel, which clients use to surface the refresh badge.
