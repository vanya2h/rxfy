# Comparison

How rxfy relates to three popular state libraries, compared against their modern,
idiomatic forms: **Redux Toolkit** (with RTK Query), **MobX 6**, and **Jotai**.

rxfy's premise is narrower than any of them: you **declare typed models and the states that
query them**, then access their data as **reactive observables**.
[Normalization](/normalization) is the storage strategy underneath, and it keeps your app
consistent and reactive at no extra cost. `data$` emits entity **ids**, values live in
per-entity Observable cells, and one `store.set` (refetch, mutation, or websocket) updates
the components subscribed to that entity and nothing else: no global re-selection, no
refetch. A `defineState` declaration bundles fetching, normalization, and SSR into one
reactive state.

## At a glance

✅ built in · ➖ possible, but you wire it up · ❌ not addressed

|                   | rxfy                                     | Redux Toolkit                     | MobX 6                    | Jotai            |
| ----------------- | ---------------------------------------- | --------------------------------- | ------------------------- | ---------------- |
| **Model**         | Reactive states over normalized entities | Immutable store + reducers        | Mutable observables       | Composable atoms |
| **Reactivity**    | Only the changed entity's subscribers    | Components whose selector output changed | Components that read the changed field | The changed atom's subscribers |
| **Data fetching** | ✅                                       | ➖ RTK Query                      | ❌                        | ➖ async atoms   |
| **Normalization** | ✅                                       | ➖ manual                         | ❌                        | ➖ manual        |
| **SSR**           | ✅                                       | ➖ manual                         | ❌                        | ➖ helpers       |
| **Validation**    | ✅ Zod                                   | ❌                                | ❌                        | ❌               |
| **Boilerplate**   | Low                                      | Moderate                          | Very low                  | Very low         |
| **Core deps**     | `rxjs`, `zod`, `lodash`                  | `@reduxjs/toolkit`, `react-redux` | `mobx`, `mobx-react-lite` | `jotai`          |

## vs Redux Toolkit

Redux centralizes state in one immutable store updated by reducers; components read through
`useSelector`, and RTK Query layers on caching and fetching. It is mature, framework-agnostic,
and has excellent time-travel devtools.

rxfy shares Redux's normalized, immutable-value approach but is **stream-first** rather than
selector-first: each entity is its own `Observable` cell, so a write notifies only that
entity's subscribers instead of re-running every selector. And where Redux composes a slice,
RTK Query, and manual SSR wiring, rxfy folds fetching + normalization + SSR into `useStateData`:

```tsx
// rxfy: define the shape once, fetch + normalize + cache + SSR come with it
const todosState = defineState({
  key: "todos",
  params: z.object({ filter: z.enum(["all", "active", "done"]) }),
  model: { todos: array(TodoModel) },
  mutations: {
    addTodo: (prev, todo: { id: string; title: string; done: boolean }) => ({
      ...prev,
      todos: [...prev.todos, todo],
    }),
  },
});

const { data$, mutations } = useStateData({ state: todosState, fetchFn: fetchTodos, params });
```

**Reach for Redux Toolkit when** you want the largest ecosystem, time-travel devtools, a
framework-agnostic single source of truth, or your state is mostly synchronous client state.
**Reach for rxfy when** normalized server data, per-entity reactive rendering, and SSR are
the core problem.

## vs MobX

MobX makes plain objects observable and lets you mutate them directly; `observer` components
re-render on the fields they read, tracked through proxies. It is the least-ceremony
option here, intuitive for local and UI state.

rxfy trades that mutable transparency for **explicit streams and immutable values**. Plain
immutable snapshots serialize and rehydrate as-is, which is central to rxfy's SSR story, where
mutable proxy graphs are harder to capture; MobX gives you no built-in normalization or
query/SSR layer anyway. Two-way binding shows the difference: MobX mutates the observable, while rxfy
composes a [`Lens`](/core-concepts/lens) over a
[`store.entity(id)`](/core-concepts/model#modelstore-entity) and drives it with
[`useAtom`](/react#useatom-and-two-way-binding), so the write flows back into the normalized
store and reaches every subscriber.

**Reach for MobX when** you want minimal boilerplate for local/UI/client state and find
direct mutation intuitive. **Reach for rxfy when** you need normalized shared server data,
reactive states, and first-class SSR.

## vs Jotai

Jotai builds state from the bottom up out of **atoms**, tiny composable units that
components subscribe to via `useAtom`. It is minimal, unopinionated, and excellent for local
and shared client state without a central store.

The shared word "atom" hides the difference. A Jotai atom is a **config object** with no value
of its own, resolved against a store (usually a React `Provider`). An rxfy
[`Atom`](/core-concepts/atom) is a real **RxJS `Observable` cell** that carries its value,
exposes synchronous `get()`/`set()`, and composes with the full operator set outside React.
Scope differs too: Jotai is a content-agnostic primitive, so normalization, validation, and a
fetch/SSR pipeline are things you assemble (often via `jotai-tanstack-query`). rxfy is
narrower and batteries-included for one job: `defineState` + `createModel` give you fetching,
normalization, Zod validation, and streaming SSR as one API.

**Reach for Jotai when** you want a minimal, flexible atomic primitive for local and shared
client state and prefer to compose data fetching and persistence yourself. **Reach for rxfy
when** normalized server data, per-entity reactive rendering, and first-class SSR are the core
problem.

## When rxfy is a good fit

- Your app is driven by **normalized server data** that several views render at once.
- You want a single live entity update (refetch, mutation, websocket) to reach every
  subscriber **without re-fetching or re-selecting**.
- **SSR with no loading flash and no hydration mismatch** matters.
- Your team is comfortable with **RxJS**, or willing to learn the small surface rxfy uses.

If your state is mostly local UI state, you need a large plugin ecosystem or time-travel
debugging, or RxJS is a non-starter for your team, Redux Toolkit or MobX will serve you
better.
